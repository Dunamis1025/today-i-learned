# Lab 06: Network Analysis — tcpdump & Wireshark

## Overview

This lab focused on two core skills in ethical hacking and network security:
1. **Capturing** network traffic using `tcpdump`
2. **Analyzing** captured traffic using `Wireshark`

The ability to capture and analyze packets is critical when performing security assessments or investigating potential network breaches.

## Part 1: Capturing Traffic with tcpdump

### What is tcpdump?
`tcpdump` is a command-line packet capture tool. It listens on a network interface and records all packets passing through it — like placing a wiretap on a network cable.

### Key Concepts

**Network Interface (eth0)**
- Every computer needs a physical or virtual network card to communicate
- `eth0` = Ethernet interface 0 — the first (and primary) network card on Kali
- All traffic between Kali and OWASP flows through `eth0`, so this is where we capture

**Protocols**
- A protocol is a set of rules that computers agree to follow when communicating — like a shared language
- **SMB (Server Message Block)**: A protocol used for network file sharing (e.g., Windows shared drives)
  - SMB1 (NT1): Very old, no encryption — all data is sent in plaintext
  - SMB2 / SMB3: Newer, more secure versions
- **HTTP (Hypertext Transfer Protocol)**: Used for web communication — also unencrypted by default

### Command Used

```bash
tcpdump -i eth0 -s0 -w testdump.pcap
```

| Flag | Meaning |
|---|---|
| `-i eth0` | Listen on the `eth0` network interface |
| `-s0` | Capture the full packet (snaplen=0 means no truncation) |
| `-w testdump.pcap` | Write output to a file instead of printing to screen |

**Why save to a file?**
- Output scrolls too fast to read in real time
- Data would be lost when the terminal closes
- `.pcap` format can be opened in Wireshark for visual analysis
- Files can be shared with teammates for collaborative investigation

**Why `-s0` (full packet capture)?**
- By default, tcpdump only saves the first ~96 bytes of each packet (just the headers)
- `-s0` ensures the entire payload is saved — critical for content analysis

### Generating Traffic to Capture

While tcpdump was running, traffic was generated by:

**1. SMB file share enumeration:**
```bash
# List available shares on OWASP server
smbclient -L 192.168.68.12 --option='client min protocol=NT1'

# Connect to the owaspbwa share
smbclient \\\\192.168.68.12\\owaspbwa --option='client min protocol=NT1'
```

- `--option='client min protocol=NT1'` forces Kali to allow the outdated SMB1 protocol (disabled by default for security reasons)
- Inside the SMB session: ran `help`, `ls`, then `exit`

**2. HTTP web browsing:**
- Opened Firefox and navigated to `192.168.68.12`
- Clicked on the **TikiWiki** link to generate HTTP page-load traffic

**Stopping the capture:**
```
Ctrl + C
```
Result: `1500 packets captured, 0 packets dropped`

## Part 2: Analyzing Traffic with Wireshark

### What is Wireshark?
Wireshark is a graphical network protocol analyzer. It reads `.pcap` files and presents packet data in a structured, human-readable format with color-coding, filtering, and protocol decoding.

### Wireshark Interface Layout

| Panel | Location | Purpose |
|---|---|---|
| Packet List | Top | One row per captured packet |
| Packet Details | Middle | Expandable protocol breakdown of selected packet |
| Packet Bytes | Bottom | Raw hexadecimal and ASCII data |

### Opening the Capture File

```bash
wireshark
```
Then: `File > Open` → select `/root/testdump.pcap`

---

### Filtering: SMB Traffic

**Filter applied:** `smb`

- 82 out of 1500 packets were SMB
- The full SMB session from the `smbclient` commands was visible:

| Packet Info | Meaning |
|---|---|
| `Negotiate Protocol Request` | Client asks: "Can we speak SMB?" |
| `Negotiate Protocol Response` | Server replies: "Yes, here are my capabilities" |
| `Session Setup AndX Request` | Login attempt (credentials transmitted) |
| `Session Setup AndX Response` | Login result |
| `Tree Connect AndX Request` | Request to connect to `\\192.168.68.12\IPC$` |

> **Security Insight:** SMB1 sends credentials and file data in plaintext. An attacker on the same network could intercept the password `owaspbwa` directly from the capture.

---

### Filtering: HTTP Traffic

**Filter applied:** `http`

- 58 out of 1500 packets were HTTP
- The Firefox browsing session was fully visible

Sample packet contents from a `GET / HTTP/1.1` request:

| HTTP Header | Value | Significance |
|---|---|---|
| `Host` | `192.168.68.12` | Target server |
| `User-Agent` | `Mozilla/5.0 ... Firefox/68.0` | Browser type and version exposed |
| `Accept-Language` | `en-US` | User's language preference leaked |

> **Security Insight:** Plain HTTP exposes everything — which pages were visited, what browser is being used, and what data was returned. HTTPS encrypts all of this.

---

### Follow TCP Stream

**How to use:** Right-click any TCP packet → `Follow > TCP Stream`

This reconstructs the full two-way conversation of a TCP connection from start to finish.

**Color coding:**
- 🔴 **Red** = Data sent by the client (Kali)
- 🔵 **Blue** = Data sent by the server (OWASP)

**What was revealed in the HTTP stream:**

```
GET / HTTP/1.1
Host: 192.168.68.12
User-Agent: Mozilla/5.0 (X11; Linux x86_64; rv:68.0) Gecko/20100101 Firefox/68.0
```

```
HTTP/1.1 200 OK
Server: Apache/2.2.14 (Ubuntu) mod_mono/2.4.3 PHP/5.3.2 ...
```

> **Critical Security Insight:** The server's response header reveals:
> - Web server: **Apache 2.2.14** (outdated, has known CVEs)
> - OS: **Ubuntu**
> - Scripting language: **PHP 5.3.2** (severely outdated)
> - These details give an attacker a precise attack surface to research known exploits against.

## Key Takeaways

### Why This Matters

| Protocol | Encrypted? | Risk |
|---|---|---|
| HTTP | ❌ No | All web content, headers, and browsing activity visible |
| SMB1 | ❌ No | Credentials and file contents visible in plaintext |
| HTTPS | ✅ Yes | Content is encrypted (only metadata like IP visible) |
| SMB3 | ✅ Yes | Encrypted with AES |

### Defensive Lessons
1. **Never use HTTP** for anything sensitive — always enforce HTTPS
2. **Disable SMB1** on all systems — use SMB2 or SMB3
3. **Remove server version headers** from HTTP responses to limit information disclosure
4. **Encrypt sensitive protocols** — assume anything unencrypted will be captured

### Attacker Perspective
- Passive sniffing (like this lab) leaves **no trace on the target**
- Tools like tcpdump and Wireshark are standard in both offensive and defensive security toolkits
- Even without active attacks, an attacker can learn enormous amounts just by listening
