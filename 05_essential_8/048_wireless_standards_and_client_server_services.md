## 1. Port Numbers

Port numbers are **logical addresses** (not physical) that identify which application/program on a device should send or receive specific data. They let a single IP address handle many simultaneous conversations.

- **Source Port** — dynamically generated by the sending device; lets it distinguish between multiple concurrent sessions (e.g., different browser tabs).
- **Destination Port** — a fixed, well-known number for a specific service on the receiving device (e.g., 80 for HTTP, 443 for HTTPS).

### Well-Known Ports Reference Table

| Port(s) | Service | Category | Description |
|---|---|---|---|
| 20, 21 | FTP | File Transfer | 20 = data, 21 = control. Insecure; SFTP/SSH preferred today. |
| 22 | SSH | Remote Access | Encrypted remote command-line access. |
| 23 | Telnet | Remote Access | Unencrypted remote access — insecure, avoid. |
| 25 | SMTP | Email | Sends/relays email between servers. |
| 53 | DNS | Web | Resolves domain names to IP addresses. |
| 67, 68 | DHCP | Network Mgmt | Auto-assigns IP addresses (server/client). |
| 69 | TFTP | File Transfer | Lightweight, simplified version of FTP. |
| 80 | HTTP | Web | Standard unencrypted web traffic. |
| 110 | POP3 | Email | Downloads email from server to local device. |
| 137–139 | NetBIOS | Network Mgmt | Legacy protocol bridging old apps to TCP/IP networks. |
| 143 | IMAP | Email | Real-time email sync across devices (stays on server). |
| 161, 162 | SNMP | Network Mgmt | Centralized monitoring/management of network devices. |
| 389 | LDAP | Identity Mgmt | Manages/authenticates directory info (usernames, credentials). |
| 427 | SLP | Network Mgmt | Auto-discovers services (printers, servers) on a LAN. |
| 443 | HTTPS | Web | Encrypted/secure web traffic. |
| 445 | SMB/CIFS | File Transfer | Shares files/printers (core of Windows networking). |
| 548 | AFP | File Transfer | Apple's proprietary file-sharing protocol. |
| 3389 | RDP | Remote Access | Microsoft's full remote desktop (GUI) control. |

## 2. TCP vs UDP — How to Tell Them Apart (Concept, Not Memorization)

The key question: does the service prioritize **reliability** (TCP) or **speed** (UDP)?

### TCP — Reliability First
Establishes a connection ("handshake") and verifies every packet arrives intact and in order.

**Used by:** File transfer (FTP, SMB, AFP), email (SMTP, POP3, IMAP), web (HTTP, HTTPS), remote access (SSH, Telnet, RDP)

**Why:** A corrupted or missing packet would break the file/email/page entirely.

### UDP — Speed First
Sends data without verifying delivery. Lost packets are simply ignored and communication continues.

**Used by:** DNS lookups, DHCP address assignment, TFTP, SNMP monitoring

**Why:** A quick query (e.g., "what's the IP for this domain?") benefits more from speed than guaranteed delivery.

### Dual-Protocol Services
Some services use **both**, switching based on context — UDP for fast/simple queries, TCP when data is larger or a stable connection is needed:
`DNS, NetBIOS, LDAP, SLP, HTTPS (QUIC/HTTP3), SMB/CIFS, RDP, AFP`

### Quick Reference

| Protocol | Logic | Examples |
|---|---|---|
| **TCP only** | Data integrity is critical | POP3, IMAP, AFP*, FTP, SSH, Telnet, SMTP |
| **UDP only** | Speed/simplicity, periodic small data | SNMP, TFTP |
| **TCP + UDP** | Flexible depending on situation | DNS, DHCP*, NetBIOS, LDAP, SLP, HTTPS, SMB, RDP |

\* *Note: exact classification (single vs. dual protocol) can vary depending on the textbook/quiz source — always check what your specific course material expects as the "official" answer.*

## 3. Wi-Fi Standards (IEEE 802.11 Series)

Each generation improves speed, range, and frequency efficiency. Legacy standards (a/b/g) are largely obsolete; **Wi-Fi 6 (802.11ax)** is the current recommendation for new deployments.

| Standard | Wi-Fi Gen | Max Speed | Range | Frequency |
|---|---|---|---|---|
| 802.11b | Wi-Fi 1 | 11 Mbps | 35 m | 2.4 GHz |
| 802.11a | Wi-Fi 2 | 54 Mbps | 35 m | 5 GHz |
| 802.11g | Wi-Fi 3 | 54 Mbps | 38 m | 2.4 GHz |
| 802.11n | Wi-Fi 4 | 600 Mbps | 70 m | 2.4/5 GHz |
| 802.11ac | Wi-Fi 5 | 6.9 Gbps | 35 m | 5 GHz |
| 802.11ax | Wi-Fi 6 | 9.6 Gbps | 46 m | 2.4/5 GHz |
| 802.11ax | Wi-Fi 6E | 9.6 Gbps | 46 m | 1/6 GHz |

> Newer standards are backward-compatible with older ones — always choose Wi-Fi 6/6E-capable hardware for new purchases.

## 4. Short-Range Wireless: Bluetooth, RFID, NFC

| Technology | Frequency | Range | Key Trait |
|---|---|---|---|
| **Bluetooth** | 2.4 GHz | Short (PAN) | Connects up to 7 devices; uses Adaptive Frequency Hopping (AFH) to reduce interference. |
| **RFID (Active)** | 125 MHz–960 MHz | Up to 100 m | Battery-powered tag; broadcasts its own ID. |
| **RFID (Passive)** | 125 MHz–960 MHz | Up to ~25 m | No battery; powered by the reader's signal. |
| **NFC** | 13.56 MHz | Very short (cm) | Subset of RFID; high security; used for contactless payments, transit cards, secure access. |

## 5. Smart Home Mesh Networks: Zigbee vs Z-Wave

| Feature | Zigbee | Z-Wave |
|---|---|---|
| Standard | IEEE 802.15.4 | Proprietary (Silicon Labs) |
| Frequency | 868 MHz – 2.4 GHz | Varies by country (e.g., 908.42 MHz in NA) |
| Range | 10–20 m | Up to 100 m |
| Speed | Faster than Z-Wave | 9.6–100 kb/s (slower) |
| Max Devices | ~65,000 | 232 |
| Requires | "Zigbee Coordinator" hub | Mesh network controller |

Both are mesh-network protocols for connecting and centrally managing many smart-home devices (usually via a smartphone app).

## 6. Cellular Network Generations

| Gen | Key Characteristic |
|---|---|
| 1G/2G | Earliest cellular tech, primarily voice. |
| 2.5G | Transitional stage — limited data/text messaging introduced. |
| 3G | Introduced real mobile internet (video calls, browsing). |
| 3.5G | Improved speed over 3G for multimedia. |
| 4G | Full mobile broadband — HD streaming, fast data. |
| LTE | The core standard powering modern 4G networks. |
| 5G | Ultra-fast, ultra-low latency, massive device connectivity (supports IoT, autonomous vehicles). Up to 3 Gb/s down / 1.5 Gb/s up. |
| 6G | Next-gen, still in R&D — aims to far exceed 5G speed and connectivity. |

## 8. Client-Server Service Flow: DHCP → DNS → HTTP

A typical sequence when a device connects to the internet and loads a website:

### Step 1 — DHCP (Dynamic Host Configuration Protocol)
A client configured for automatic addressing sends a **DHCP request**. The DHCP server (often a local router) responds with:
- An IP address for the client
- The default gateway address
- A DNS server address

### Step 2 — DNS (Domain Name System)
The client wants to reach `www.netacad.com` but needs its IP address first. It sends a **DNS request** to a DNS server, which resolves the domain name and returns the correct IP address.

### Step 3 — HTTP (HyperText Transfer Protocol)
With the IP address known, the client sends an **HTTP request** to the web server. The server responds with the requested files (HTML, images, audio, video), which the browser renders as the webpage.

**Flow summary:**
```
Client → DHCP Server   : "Give me an IP address"
DHCP Server → Client   : IP + Gateway + DNS server info
Client → DNS Server    : "What's the IP for www.netacad.com?"
DNS Server → Client    : Resolved IP address
Client → Web Server    : HTTP request (using resolved IP)
Web Server → Client    : Webpage files (HTML, images, etc.)
```

## Key Takeaways

1. **Ports** identify *which application* handles incoming/outgoing data — think of them as numbered doors.
2. **TCP vs UDP** comes down to reliability vs. speed — don't memorize blindly, reason from the service's purpose.
3. **Wi-Fi standards (802.11)** have evolved for higher speed/range; Wi-Fi 6/6E is current best practice.
4. **Short-range wireless** (Bluetooth, RFID, NFC) each trade off range vs. security vs. power needs.
5. **Smart home protocols** (Zigbee, Z-Wave) are mesh networks built for scale (Zigbee) vs. range (Z-Wave).
6. **Cellular generations** trace a path from voice-only (1G) to ultra-low-latency IoT-ready networks (5G/6G).
7. **DHCP → DNS → HTTP** is the foundational request flow behind nearly every web interaction.
